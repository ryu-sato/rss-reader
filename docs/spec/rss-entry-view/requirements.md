# RSSエントリー閲覧 要件定義書

## 概要

RSSリーダーWebアプリケーションにおいて、登録済みRSSフィードのエントリー（記事）を閲覧する機能。
全フィードのエントリーをまとめて時系列降順（新しい順）で表示し、モーダルで詳細閲覧・メタ情報管理を行う。
既読管理・後で読む・タグ付けによるパーソナライズ、フィード別・タグ別フィルタリングを提供する。
バックグラウンドで1時間ごとに全フィードのエントリーを自動取得・更新する。

## 機能要件（EARS記法）

**【信頼性レベル凡例】**:
- 🔵 **青信号**: PRD・EARS要件定義書・設計文書・ユーザヒアリングを参考にした確実な要件
- 🟡 **黄信号**: PRD・EARS要件定義書・設計文書・ユーザヒアリングから妥当な推測による要件
- 🔴 **赤信号**: PRD・EARS要件定義書・設計文書・ユーザヒアリングにない推測による要件

### 通常要件

- REQ-001: システムは全フィードのエントリーを公開日時降順（新しい順）で一覧表示しなければならない 🔵 *ヒアリング: 全フィードまとめ読み・新しい順より*
- REQ-002: システムはエントリー一覧にエントリータイトルとフィード名を表示しなければならない 🔵 *ヒアリング: 表示項目（タイトル・フィード名）より*
- REQ-003: システムはエントリー一覧を1ページ20件でページネーション表示しなければならない 🔵 *ヒアリング: 20件・ページネーション必要より*
- REQ-004: システムはエントリー一覧をフィード別にフィルタリングできなければならない 🔵 *ヒアリング: フィルター（両方）より*
- REQ-005: システムはエントリー一覧をタグ別にフィルタリングできなければならない 🔵 *ヒアリング: フィルター（両方）より*
- REQ-006: システムはエントリーのタイトル・公開日時・全文・元記事リンク・メタ情報をモーダルで表示しなければならない 🔵 *ヒアリング: モーダル表示内容より*
- REQ-007: システムはモーダルに前後エントリーへの移動ボタンを提供しなければならない 🔵 *ヒアリング: 前後モーダル移動ボタンより*
- REQ-008: システムは各エントリーの既読・未読状態を管理しなければならない 🔵 *ヒアリング: 既読管理 必要より*
- REQ-009: システムは各エントリーに「後で読む」フラグを設定・解除できなければならない 🔵 *ヒアリング: モーダルの既読・後で読む・タグより*
- REQ-010: システムは各エントリーにタグを付与・削除できなければならない 🔵 *ヒアリング: タグ機能より*
- REQ-011: システムはタグを自由入力で新規作成できなければならない 🔵 *ヒアリング: 未登録タグは自由入力により作成より*
- REQ-012: システムは登録済みタグをエントリーに対して選択付与できなければならない 🔵 *ヒアリング: 登録済みのタグは選択可能より*
- REQ-013: システムはRSSフィードのエントリーをDBにタイトル・リンク・概要・全文・公開日時を含めて保存しなければならない 🔵 *ヒアリング: 全文も保存より*

### 条件付き要件

- REQ-101: エントリーをモーダルで開いた場合、システムはそのエントリーを自動的に既読状態にしなければならない 🟡 *既読管理の一般的な動作から妥当な推測*
- REQ-102: フィードのエントリーが500件を超えた場合、システムは古いエントリーから削除しなければならない 🔵 *ヒアリング: 上限あり・500件より*
- REQ-103: 新規エントリーを取得した場合、システムはDBに保存済みの同一GUIDのエントリーを重複登録してはならない 🟡 *重複排除の標準的な要件として推測*
- REQ-104: エントリー取得に失敗した場合（ネットワークエラー等）、システムは既存DBのエントリーを保持し、取得失敗をログに記録しなければならない 🟡 *堅牢性要件として妥当な推測*
- REQ-105: エントリーに既読メタ情報がない場合、システムはそのエントリーを未読として扱わなければならない 🟡 *既読管理の実装として妥当な推測*

### 状態要件

- REQ-201: エントリー一覧が0件の状態にある場合、システムはフィードを追加するよう促すメッセージを表示しなければならない 🟡 *UX要件として妥当な推測*
- REQ-202: エントリー取得中の状態にある場合、システムはローディング表示を行いユーザーに処理中であることを伝えなければならない 🟡 *UX要件として妥当な推測*

### 状態要件（定期取得）

- REQ-211: バックグラウンドで1時間ごとに全フィードのエントリーを自動取得・DB更新しなければならない 🔵 *ヒアリング: 定期自動取得・1時間ごとより*

### オプション要件

- REQ-301: システムはエントリー一覧で未読件数を表示してもよい 🟡 *UX要件として妥当な推測*

### 制約要件

- REQ-401: システムは1フィードあたりのエントリー保存件数を500件に制限しなければならない 🔵 *ヒアリング: 500件より*
- REQ-402: システムはユーザー認証なしで全エントリーを共通として管理しなければならない 🔵 *既存フィード登録要件 REQ-404から継承*
- REQ-403: システムはエントリーの取得・保存をバックグラウンドで行い、閲覧に影響を与えてはならない 🟡 *UX要件として妥当な推測*

## データモデル

### エントリー（Entry）

| フィールド | 型 | 説明 | 必須 |
|-----------|-----|------|------|
| id | string (UUID) | 一意識別子 | ✅ |
| feedId | string (UUID, FK) | フィードID | ✅ |
| guid | string | RSS エントリーの GUID | ✅ |
| title | string | エントリータイトル | ✅ |
| link | string | 元記事URL | ✅ |
| description | string \| null | 概要（summary） | - |
| content | string \| null | 全文（content） | - |
| publishedAt | datetime \| null | 公開日時 | - |
| createdAt | datetime | DB登録日時 | ✅ |
| updatedAt | datetime | 最終更新日時 | ✅ |

🔵 *ヒアリング: 全文も保存・表示内容より*

### エントリーメタ情報（EntryMeta）

| フィールド | 型 | 説明 | 必須 |
|-----------|-----|------|------|
| id | string (UUID) | 一意識別子 | ✅ |
| entryId | string (UUID, FK, unique) | エントリーID | ✅ |
| isRead | boolean | 既読フラグ（default: false） | ✅ |
| isReadLater | boolean | 後で読むフラグ（default: false） | ✅ |
| createdAt | datetime | 作成日時 | ✅ |
| updatedAt | datetime | 最終更新日時 | ✅ |

🔵 *ヒアリング: 既読・後で読む管理より*

### タグ（Tag）

| フィールド | 型 | 説明 | 必須 |
|-----------|-----|------|------|
| id | string (UUID) | 一意識別子 | ✅ |
| name | string (unique) | タグ名 | ✅ |
| createdAt | datetime | 作成日時 | ✅ |

🔵 *ヒアリング: タグ機能より*

### エントリータグ（EntryTag, 中間テーブル）

| フィールド | 型 | 説明 | 必須 |
|-----------|-----|------|------|
| entryId | string (UUID, FK) | エントリーID | ✅ |
| tagId | string (UUID, FK) | タグID | ✅ |

複合主キー: (entryId, tagId) 🟡 *標準的なM:N設計から推測*

## 非機能要件

### パフォーマンス

- NFR-001: エントリー一覧の表示は2秒以内に応答しなければならない 🟡 *Web UX基準から妥当な推測*
- NFR-002: 定期取得処理は1回あたり5分以内に完了しなければならない 🟡 *バッチ処理の一般的な要件から推測*
- NFR-003: 各フィードの取得タイムアウトは30秒とする（既存設定 NFR-001 と同様） 🔵 *既存フィード登録 NFR-001 から継承*

### セキュリティ

- NFR-101: エントリーのHTML全文を表示する場合、XSS対策のサニタイズを行わなければならない 🟡 *HTMLコンテンツ表示の一般的なセキュリティ要件から推測*
- NFR-102: 定期取得処理も既存の SSRF 対策（ssrf-guard）を適用しなければならない 🔵 *既存 NFR-101 から継承*

### ユーザビリティ

- NFR-201: エントリーの既読・後で読む・タグ設定はモーダルから即時反映されなければならない 🔵 *ヒアリング: モーダルで設定より*
- NFR-202: エラーメッセージは英語で表示しなければならない 🔵 *既存実装（architecture.md）より*

## エッジケース

### エラー処理

- EDGE-001: エントリーの全文（content）がない場合、概要（description）を全文として代用する 🟡 *RSS仕様上の一般的な対応から推測*
- EDGE-002: エントリーの公開日時（publishedAt）がない場合、DB登録日時（createdAt）を表示日時として使用する 🟡 *RSS仕様上の一般的な対応から推測*
- EDGE-003: エントリーのタイトルが空の場合、リンクURLをフォールバックタイトルとして使用する 🟡 *既存フィード登録 EDGE-003 と同様の対応*
- EDGE-004: 定期取得で一部フィードが失敗しても、他のフィードの取得を継続しなければならない 🟡 *堅牢性要件として妥当な推測*

### 境界値

- EDGE-101: エントリー本文（content）が非常に長い場合（例: 100,000文字超）、モーダル表示時にスクロール可能な形式で表示する 🟡 *UI要件として妥当な推測*
- EDGE-102: タグ名が重複している場合（大文字小文字の違いを含む）、同一タグとして扱うか否かを定義する必要がある 🔴 *推測（要確認）*
- EDGE-103: フィルターでフィード・タグを複数選択した場合の動作（AND/OR）を定義する 🔴 *推測（要確認）*
