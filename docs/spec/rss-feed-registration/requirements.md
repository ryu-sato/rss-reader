# RSSシードの登録 要件定義書

## 概要

RSSリーダーWebアプリケーションにおいて、ユーザーがRSSフィードのURLを登録・管理する機能。
Next.js（フルスタック）+ SQLiteを使用し、RSS URL入力→検証→フィード情報自動取得→保存のフローを提供する。
認証なしの共通リスト管理。

## 機能要件（EARS記法）

**【信頼性レベル凡例】**:
- 🔵 **青信号**: ユーザヒアリングを参考にした確実な要件
- 🟡 **黄信号**: ユーザヒアリングから妥当な推測による要件
- 🔴 **赤信号**: ユーザヒアリングにない推測による要件

### 通常要件

- REQ-001: システムはRSSフィードURLを入力するフォームを提供しなければならない 🔵 *ヒアリングQ1: URLを入力して登録を選択*
- REQ-002: システムは登録済みRSSフィードの一覧を表示しなければならない 🔵 *ヒアリングQ1: フィードの一覧表示より*
- REQ-003: システムはフィード一覧にフィードタイトル・登録日時・最終更新日時を表示しなければならない 🔵 *ヒアリングQ7: 一覧表示項目より*
- REQ-004: システムは各フィードにメモを保存・表示しなければならない 🔵 *ヒアリングQ5: ユーザー定義のメモを選択*

### 条件付き要件

- REQ-101: ユーザーがRSS URLを入力して登録操作を行った場合、システムは実際にそのURLへHTTPリクエストを送信しRSSとして有効か検証しなければならない 🔵 *ヒアリングQ6: 実際に取得して検証を選択*
- REQ-102: URL検証が成功した場合、システムはフィードのタイトルと説明を自動取得して登録データとして保存しなければならない 🔵 *ヒアリングQ1: フィード情報の自動取得を選択*
- REQ-103: 入力されたURLが既に登録済みである場合、システムはエラーメッセージを表示し登録を拒否しなければならない 🔵 *ヒアリングQ8: 重複登録不可を選択*
- REQ-104: ユーザーがフィードの編集操作を行った場合、システムはフィード情報（タイトル・説明・メモ）を更新できる画面を提供しなければならない 🔵 *ヒアリングQ1: フィードの編集・削除を選択*
- REQ-105: ユーザーがフィードの削除操作を行った場合、システムは確認ダイアログを表示し、確認後にフィードを削除しなければならない 🔵 *ヒアリングQ9: 確認ダイアログあり*
- REQ-106: 入力されたURLがHTTPリクエストに失敗した場合、システムはエラーメッセージを表示し登録を中断しなければならない 🔵 *ヒアリングQ6: 実際に取得して検証より派生*
- REQ-107: 取得したコンテンツが有効なRSS/Atom形式でない場合、システムはエラーメッセージを表示し登録を中断しなければならない 🟡 *URL検証要件から妥当な推測*

### 状態要件

- REQ-201: URL検証中の状態にある場合、システムはローディング表示を行いユーザーに処理中であることを伝えなければならない 🟡 *UX要件として妥当な推測*
- REQ-202: フィードが1件も登録されていない状態にある場合、システムはフィード登録を促すメッセージを表示しなければならない 🟡 *UX要件として妥当な推測*

### 制約要件

- REQ-401: システムは同一URLのフィードを複数登録してはならない 🔵 *ヒアリングQ8: 重複登録不可*
- REQ-402: システムはフィードURLをhttps://またはhttp://から始まる形式に制限しなければならない 🟡 *URL検証の前提条件として妥当な推測*
- REQ-403: システムはフィード登録件数に上限を設けてはならない 🔵 *ヒアリングQ4: 上限なし*
- REQ-404: システムはユーザー認証なしで全フィードを共通リストとして管理しなければならない 🔵 *ヒアリングQ3: 認証不要*

## データモデル

### フィード（Feed）

| フィールド | 型 | 説明 | 必須 |
|-----------|-----|------|------|
| id | string (UUID) | 一意識別子 | ✅ |
| url | string (unique) | RSSフィードURL | ✅ |
| title | string | フィードタイトル（自動取得） | ✅ |
| description | string \| null | フィード説明（自動取得） | - |
| memo | string \| null | ユーザー定義メモ | - |
| createdAt | datetime | 登録日時 | ✅ |
| updatedAt | datetime | 最終更新日時 | ✅ |
| lastFetchedAt | datetime \| null | 最終フェッチ日時 | - |

🔵 *ヒアリングQ5: 保存情報選択より*

## 非機能要件

### パフォーマンス

- NFR-001: URLの検証・フィード情報取得は30秒以内にタイムアウトしなければならない 🟡 *一般的なHTTPタイムアウト値から妥当な推測*
- NFR-002: フィード一覧の表示は1秒以内に応答しなければならない 🟡 *Web UX基準から妥当な推測*

### セキュリティ

- NFR-101: システムはSSRF（Server-Side Request Forgery）対策として、プライベートIPアドレスへのリクエストを拒否しなければならない 🔴 *セキュリティ要件として推測（要確認）*
- NFR-102: システムは入力URLをサニタイズしてSQLインジェクションを防止しなければならない 🟡 *一般的なセキュリティ要件として推測*

### ユーザビリティ

- NFR-201: URL入力フォームはリアルタイムで基本的なURL形式バリデーションを行いフィードバックを提供しなければならない 🟡 *UX要件として妥当な推測*
- NFR-202: エラーメッセージは日本語で表示しなければならない 🔴 *プロジェクト名から推測（要確認）*

## エッジケース

### エラー処理

- EDGE-001: ネットワーク接続エラーが発生した場合、適切なエラーメッセージを表示する 🟡 *一般的なエラー処理から推測*
- EDGE-002: タイムアウトが発生した場合、タイムアウト専用のメッセージを表示する 🟡 *NFR-001から導出*
- EDGE-003: RSSフィードのタイトルが空の場合、URLをフォールバックとしてタイトルに使用する 🟡 *一般的な処理から推測*
- EDGE-004: フィード説明が非常に長い場合（1000文字超）、適切に切り詰めて保存する 🔴 *推測（要確認）*

### 境界値

- EDGE-101: URLが2048文字を超える場合、入力を拒否しエラーを表示する 🟡 *URL長の一般的上限から推測*
- EDGE-102: メモが1000文字を超える場合の挙動を定義する必要がある 🔴 *推測（要確認）*
